تعليمات بناء متجر ملابس ولوحة تحكم كاملة
========================================

هذا الملف تعليمات شاملة يمكن إعطاؤها لأي ذكاء اصطناعي مع لوجو واسم براند جديد لبناء موقع مشابه. يجب أن تكون التعليمات عامة وغير مرتبطة بلوجو محدد. عند تغيير اللوجو أو اسم البراند، يجب استخراج الهوية البصرية الجديدة من اللوجو والمدخلات الجديدة.

الفكرة العامة
-------------
ابن موقعا ثابتا Static Website مناسبا للرفع على GitHub Pages لبراند ملابس، مكونا من:

- متجر للعملاء.
- لوحة تحكم للأدمن.
- ملفات HTML وCSS وJavaScript فقط.
- تخزين محلي باستخدام localStorage لأن GitHub Pages لا يوفر باك إند.
- صفحات منفصلة، وليس كل الأقسام داخل صفحة واحدة.
- دعم العربية RTL كأساس.
- دعم الإنجليزية LTR.
- زر تبديل لغة عربي/إنجليزي.
- زر تبديل الوضع النهاري والليلي.
- تصميم متجاوب للموبايل والتابلت والديسكتوب.
- الدفع عند الاستلام فقط.
- تنبيه الطلب عبر واتساب الأدمن.
- تنبيه الطلب عبر EmailJS عند تفعيله.

مهم:
هذا النظام مناسب كبداية Static على GitHub Pages. إذا تحول المشروع لتجارة كبيرة، يجب بناء باك إند وقاعدة بيانات ونظام صلاحيات حقيقي.

الهوية البصرية
--------------
عند استلام اللوجو:

- استخرج الألوان الرئيسية والثانوية منه.
- ابن لوحة ألوان راقية تناسب براند ملابس.
- استخدم لون Accent واضح للأزرار والعناصر المهمة.
- لا تجعل الموقع كله بدرجة لون واحدة.
- اجعل اللوجو واضحا في الهيدر وصفحة الأدمن والفاتورة والـ favicon.
- صمم الوضع الليلي بعناية، وليس مجرد خلفية سوداء ونص أبيض.
- اجعل الخطاب مناسبا للرجال والنساء والأطفال، وليس موجها للنساء فقط.
- لا تكتب نصوصا ثابتة مخصوصة بلوجو أو براند معين.

الصفحة الرئيسية للعملاء
----------------------
الصفحة الرئيسية يجب أن تبدأ بتجربة عملية وليس صفحة تسويقية طويلة.

يجب أن تحتوي على:

- هيدر فيه اللوجو.
- زر القائمة الجانبية.
- زر اللغة.
- زر الوضع الليلي/النهاري.
- زر السلة.
- 3 اختيارات رئيسية كبيرة: رجال، نساء، أطفال.
- كل اختيار بصورة أو خلفية واضحة ومناسبة.
- عند الضغط على أي اختيار ينتقل العميل إلى صفحة القسم.
- أسفل الاختيارات تظهر مزايا حقيقية للعميل، مثل:
  - خامات مختارة.
  - سعر مناسب مقابل الجودة.
  - قطع سهلة التنسيق.
  - دفع عند الاستلام.
  - تواصل واضح عبر واتساب.

لا تضع داخل موقع العميل كلاما يخص الأدمن مثل جودة الفاتورة أو المخزون أو لوحة التحكم.

القائمة الجانبية للعملاء

- أطفال.

EmailJS
-------
لأن GitHub Pages لا يوفر باك إند، استخدم EmailJS لإرسال الإيميل من المتصفح.

يجب أن توجد إعدادات EmailJS في مكانين:

1. لوحة الأدمن حتى يقدر الأدمن يختبر ويعدل القيم من متصفحه.
2. ملف البيانات الأساسي assets/glitter-data.js كقيم افتراضية، لأن إعدادات localStorage لا تنتقل لكل العملاء.

مهم جدا:
لو الأدمن حط مفاتيح EmailJS من لوحة الأدمن فقط، فهي محفوظة في متصفحه هو فقط. العميل من جهاز آخر لن يراها. لذلك يجب وضع القيم الافتراضية أيضا في data.js قبل رفع الموقع.

إعدادات EmailJS:

- emailjsEnabled.
- emailjsServiceId.
- emailjsTemplateId.
- emailjsPublicKey.
- supportEmail أو adminEmail.

مثال:

emailjsEnabled: true
emailjsServiceId: "service_xxxxx"
emailjsTemplateId: "template_xxxxx"
emailjsPublicKey: "xxxxxxxx"

Template ID
-----------
الـ Template ID الثابت يؤخذ من صفحة القالب في EmailJS.

لو رابط القالب مثل:

https://dashboard.emailjs.com/admin/templates/iwbybdw

فالجزء المهم هو:

iwbybdw

لكن بعض إعدادات EmailJS تستخدم الصيغة:

template_iwbybdw

لذلك الأفضل أن يدعم الكود الصيغتين:

- iwbybdw.
- template_iwbybdw.

لو ظهرت رسالة:

The template ID not found

فالأسباب المحتملة:

- Template ID غير صحيح.
- Service ID وTemplate ID من حسابين مختلفين.
- Public Key من حساب مختلف.
- المتصفح يحتفظ بقيمة قديمة في localStorage.
- الملف الجديد لم يتم رفعه أو الكاش لم يتحدث.

يجب إضافة زر "اختبار EmailJS" داخل إعدادات الأدمن. هذا الزر يستخدم نفس مفاتيح الموقع ونفس القالب ويرسل رسالة اختبار حقيقية.

متغيرات قالب EmailJS
--------------------
اجعل الموقع يرسل المتغيرات التالية:

- name.
- email.
- reply_to.
- time.
- to_email.
- admin_email.
- store_name.
- order_id.
- customer_name.
- customer_phone.
- customer_city.
- customer_area.
- customer_address.
- customer_notes.
- order_items.
- order_subtotal.
- order_shipping.
- order_total.
- payment_method.
- message.

في EmailJS يمكن استخدام:

{{order_id}}
{{customer_name}}
{{customer_phone}}
{{customer_address}}
{{order_items}}
{{order_total}}
{{message}}

خانة Reply To يفضل أن تكون:

{{reply_to}}

لو لم يتم تفعيل EmailJS أو كانت القيم ناقصة، لا يتعطل الطلب. يسجل الطلب ويفتح واتساب فقط.

لو فشل EmailJS، اطبع الخطأ في Console بوضوح.

لو كان السبب Template ID not found، جرّب تلقائيا الصيغة البديلة بإضافة أو إزالة template_.

لو احتفظ المتصفح بقيم قديمة:

- احفظ الإعدادات من جديد.
- أو امسح localStorage.
- أو اجعل seed/migration في data.js يستبدل القيم القديمة الخاطئة بالقيم الافتراضية الجديدة.

للحماية، استخدم Authorized Domains داخل EmailJS واجعل الدومين المسموح هو دومين الموقع.

التعديل البصري
--------------

لا تجعل التعديل البصري مقتصرا على الصفحة الرئيسية فقط.

أضف اختيار صفحة:

- الرئيسية.
- رجال.
- نساء.
- أطفال.
- كل المنتجات إن وجدت.

في الصفحة الرئيسية يمكن تعديل:

- عنوان الهيرو.
- وصف الهيرو.
- كروت الأقسام.
- صورة كل قسم.
- اسم كل قسم.
- وصف كل قسم.
- ترتيب الأقسام.
- بطاقات المميزات.

داخل صفحات الأقسام يمكن تعديل:

- هيرو القسم.
- اسم القسم.
- وصف القسم.
- صورة القسم.

GitHub Pages
------------
لرفع الموقع:

- اجعل index.html هي الصفحة الرئيسية.
- استخدم روابط نسبية.
- أضف .nojekyll.

ملف CNAME يحتوي على الدومين فقط:

example.com

لا تكتب:

https://example.com/

بعد تغيير CSS أو JS، قد يحتاج المستخدم Ctrl + F5.

لو HTTPS لم يعمل فورا، انتظر إصدار شهادة GitHub Pages ثم فعل Enforce HTTPS.

- meta title مختلف لكل صفحة مهمة.
- meta description مختلف لكل صفحة مهمة.

Sitemap: https://yourdomain.com/sitemap.xml

sitemap.xml
-----------
يجب أن يحتوي على الصفحات التي نريد ظهورها فقط:

- الصفحة الرئيسية.
- صفحة الرجال.
- صفحة النساء.
- صفحة الأطفال.
- صفحة المعلومات.

لا تضف صفحات الأدمن.
لا تضف السلة والدفع.

مثال:

https://yourdomain.com/
https://yourdomain.com/glitter-men.html
https://yourdomain.com/glitter-women.html
https://yourdomain.com/glitter-kids.html
https://yourdomain.com/glitter-info.html

بعد رفع الملفات، يجب أن تعمل الروابط:

https://yourdomain.com/robots.txt
https://yourdomain.com/sitemap.xml

Google Search Console
---------------------
خطوات Google Search Console:

1. افتح:
   https://search.google.com/search-console
2. أضف Domain property.
3. اكتب الدومين بدون https.
4. انسخ TXT verification record.
5. افتح DNS في مزود الدومين.
6. أضف TXT record:
   Name/Host: @
   Value: google-site-verification=...
7. انتظر من 5 دقائق إلى 24 ساعة.
8. ارجع لجوجل واضغط Verify.

إرسال sitemap
-------------
لو الحساب Domain property، أرسل الرابط الكامل:

https://yourdomain.com/sitemap.xml

لو ظهر خطأ "Site map title is invalid"، غالبا كتبت sitemap.xml فقط في Domain property. اكتب الرابط الكامل.

لو ظهر "The file could not be retrieved" مع أن الرابط يفتح في المتصفح:

- احذف السطر القديم إن أمكن.
- أعد إرسال الرابط الكامل.
- انتظر من 30 دقيقة إلى 24 ساعة.
- تأكد أن sitemap.xml في جذر الموقع بجانب index.html وليس داخل assets.

فحص URL وطلب الفهرسة
-------------------
من URL Inspection افحص:

https://yourdomain.com/

ثم اضغط Request indexing.

كرر لأهم الصفحات:

- /glitter-men.html.
- /glitter-women.html.
- /glitter-kids.html.
- /glitter-info.html.

لا تطلب فهرسة صفحات الأدمن أو السلة.

معنى "URL is available on Google"
--------------------------------
لو Search Console كتب:

URL is available on Google

أو:

تمت فهرسة الصفحة

فهذا يعني أن جوجل قبل الصفحة وفهرسها.

لكن الفهرسة لا تعني أنها ستظهر فورا عند البحث العام. يوجد فرق بين:

- الفهرسة: جوجل سجل الصفحة.
- الظهور: جوجل يعرضها في نتائج البحث.
- الترتيب: الصفحة تظهر في نتيجة قريبة أو بعيدة.

بعد الفهرسة قد تتأخر نتائج البحث العامة من 24 إلى 72 ساعة أو أكثر.

ماذا تبحث بعد الفهرسة؟
---------------------
استخدم:

site:yourdomain.com

مثال:

site:glitterstore.online

ثم ابحث باسم الدومين:

"glitterstore.online"

ثم اسم البراند:

"Glitter Store"
"Glitter Fashion"
"Glitter GL Fashion"

وللصفحات:

site:glitterstore.online glitter men
site:glitterstore.online glitter women
site:glitterstore.online glitter kids

باللغة العربية:

Glitter ملابس
Glitter ملابس رجالي
Glitter ملابس حريمي
Glitter ملابس أطفال
Glitter Store ملابس

مهم:
الظهور باسم البراند والدومين أسرع. الظهور في كلمات عامة مثل "ملابس رجالي" يحتاج وقتا ومحتوى وروابط خارجية ومنافسة.

Structured Data
---------------
للمتاجر، الأفضل مستقبلا إضافة Product structured data.

لكن بما أن المنتجات حاليا تظهر من JavaScript ولا توجد صفحة مستقلة لكل منتج، الفائدة محدودة.

الأفضل لاحقا:

- إنشاء صفحة مستقلة لكل منتج.
- إضافة JSON-LD لكل منتج.
- إدخال name وimage وdescription وoffers وprice وavailability.

التعامل مع localStorage
-----------------------
لأن المشروع Static:

- المنتجات تحفظ في localStorage.
- الطلبات تحفظ في localStorage.
- السلة تحفظ في localStorage.
- الإعدادات تحفظ في localStorage.

لكن أي تعديل من لوحة الأدمن على جهاز معين لا يظهر تلقائيا لكل الزوار إلا إذا كانت البيانات جزءا من الملف المرفوع أو يوجد باك إند.

لذلك:

- القيم العامة المهمة مثل EmailJS وبيانات التواصل الأساسية يجب أن تكون في data.js قبل الرفع.
- التعديلات المحلية من الأدمن تخص المتصفح الحالي فقط.
